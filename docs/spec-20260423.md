# wcmux — Web 版 cmux 需求文档

> 基于 FastAPI 的浏览器端终端多路复用器，支持多 Tab、密码登录，可配置 `base_url` 与 `port`。

## 1. 总体目标

**需求**: 提供一个通过浏览器访问的 cmux/tmux 风格终端工具，用户登录后可在同一页面打开多个终端 Tab，每个 Tab 是一路独立的 shell。
**动机**: 在没有 SSH 客户端或受限网络环境下，也能通过浏览器获得多会话终端；相比只能开单个终端的 Web SSH，Tab 结构让日常多任务运维更顺手。
**非目标**:
- 不实现 tmux 原生的 detach/attach/分屏（pane split）/复制粘贴缓冲区，仅做多 Tab。
- 不支持多用户并发、权限分级；当前只有一套密码、一个用户。
- 不做文件上传下载、不做图形转发。

---

## 2. 运行配置

### 2.1. 启动参数与环境变量

**需求**: 程序启动时可通过命令行参数或环境变量配置以下几项：
- `--port` / `WCMUX_PORT`：HTTP 监听端口，默认 `8022`。
- `--base-url` / `WCMUX_BASE_URL`：部署路径前缀，默认 `/`；用于反向代理下挂载到子路径（如 `/wcmux`）。
- `--password` / `WCMUX_PASSWORD`：登录密码，**必填**，未设置则启动失败并打印明确错误。
- `--shell` / `WCMUX_SHELL`：新 Tab 启动时执行的 shell 可执行文件绝对路径；未设置时按平台选默认：Unix 优先 `$SHELL`，回退 `/bin/bash`；Windows 优先 `%COMSPEC%`，回退 `powershell.exe`。

**约束**:
- 命令行参数优先级高于环境变量。
- `base_url` 规范化：自动补前导 `/`、去掉末尾 `/`（空字符串或 `/` 视为根路径）。
- 所有路由、静态资源、WebSocket 连接地址都必须尊重 `base_url` 前缀；前端页面能在子路径下正常加载。
- `port` 取值范围 `1–65535`，非法值启动失败。

**非目标**: 不支持热加载配置，改了要重启。

### 2.2. 监听地址

**需求**: 默认监听 `0.0.0.0`；可通过 `--host` / `WCMUX_HOST` 覆盖。
**动机**: 默认允许 LAN 访问以便实际使用；但保留回退到 `127.0.0.1` 的能力。

## 3. 密码登录认证

### 3.1. 登录入口

**需求**: 未登录用户访问任何页面/接口都会被重定向到登录页 `/login`（考虑 `base_url` 前缀），登录页提供密码输入框与提交按钮。
**约束**:
- 密码通过 POST `application/x-www-form-urlencoded` 或 JSON 提交，不在 URL query 中传输。
- 密码错误时返回 401 并在页面提示"密码错误"；成功时 set session cookie 并重定向回原目标。

### 3.2. 会话管理

**需求**: 使用服务端签名的 session cookie 记录登录状态，默认有效期 7 天；提供登出入口清除 cookie。
**约束**:
- Cookie 属性必须包含 `HttpOnly`、`SameSite=Lax`；当请求协议为 HTTPS 时自动加 `Secure`。
- Session 秘钥 `--secret-key` / `WCMUX_SECRET_KEY` 可配置；未配置则启动时生成随机 key 并打印警告（重启后 session 失效）。
- WebSocket 握手时必须校验 session cookie，未登录直接拒绝握手（HTTP 401/403）。

### 3.3. 暴力破解保护（基础）

**需求**: 同一 IP 连续 5 次密码错误后，锁定 15 分钟内拒绝所有来自该 IP 的登录尝试。
**动机**: 部署在公网时的最小防护。
**约束**: 锁定状态仅内存保存，进程重启清零；足够作为 L1 防护，不追求严格审计。

### 3.4. 密码加密存储

**需求**: 服务端不以明文形式比对或持久化登录密码；配置接受已加盐的哈希值，运行时用常量时间比较验证；仅提供明文入口时，启动阶段立刻派生哈希并丢弃明文引用。
**动机**: 避免 `ps aux`、systemd `EnvironmentFile`、core dump、日志或环境变量抓取等途径把明文密码泄露给攻击者；也避免配置被 `cat` 误发出去时直接暴露可登录凭证。
**约束**:
- 新增 `--password-hash` / `WCMUX_PASSWORD_HASH`，优先级**高于**明文 `--password` / `WCMUX_PASSWORD`。
- 支持的哈希格式按前缀识别：`$argon2id$...`（推荐）、`$argon2i$...`、`$2a$...` / `$2b$...` / `$2y$...`（bcrypt）。其他前缀启动失败并打印明确错误，不 fallback 到明文。
- 明文入口保留为向后兼容：启动时立即算一次 argon2id 哈希替换使用，**立即解除对明文字符串的引用**（尽 Python 可能），此后该明文不再出现在日志、`repr`、`/proc/<pid>/environ`（配合启动脚本建议用 `unset` 或 `systemd-run --setenv` 的一次性注入）、异常堆栈。
- 同时提供哈希与明文时，以哈希为准、明文被忽略并打印 warning `plaintext password ignored because hash is set`。
- 仅提供明文时启动打印一次 warning `consider providing --password-hash instead of plaintext`。
- 增加子命令 `wcmux hash-password`（`python -m wcmux hash-password`）：从 stdin / 提示符读入密码（不回显），生成 argon2id 哈希打印到 stdout，不写磁盘。
- 哈希校验失败仍按 §3.3 计入 IP 锁定。
- 哈希算法选择由服务端决定（当前 argon2id 参数：`m=64MB, t=3, p=4`），用户只负责给一个合法的哈希字符串，不暴露可调的算法参数。

**非目标**: 不做多用户 / 多密码；不做 API Token；不提供密码迁移工具（变密码就重生成哈希）；不加 HSM / KMS 集成。

## 4. 终端会话与多 Tab

### 4.1. 新建 Tab

**需求**: 页面顶部有"+"按钮，点击后在当前页面新增一个 Tab，并为其在服务端 fork 一路新的 shell 进程（默认 `$SHELL`，fallback `/bin/bash`）。
**约束**:
- 每个 Tab 通过独立的 WebSocket 通道与后端 PTY 双向通信。
- 新建 Tab 时自动切到该 Tab；Tab 名默认 `Shell N`（N 为序号），支持双击重命名。
- 单浏览器页面最多同时存在 **20** 个 Tab，超出时"+"按钮禁用并提示上限。

### 4.2. 切换 Tab

**需求**: 点击 Tab 标签切换到对应终端；未激活的 Tab 在后台保持运行、继续接收输出。
**约束**:
- 切换 Tab 不中断后台 Tab 的 shell 进程。
- 后台 Tab 有新输出时，Tab 标签显示未读提示（如加粗或圆点），切到该 Tab 后清除。

### 4.3. 关闭 Tab

**需求**: 每个 Tab 标签上有关闭按钮；关闭时终止对应 shell 进程（SIGHUP），回收 PTY。
**约束**:
- 关闭最后一个 Tab 后自动打开一个新的空 Tab，不让页面变空。
- 关闭时如有未保存内容，不做额外确认（与本地终端一致）。

### 4.4. 终端渲染与尺寸

**需求**: 前端使用 xterm.js 渲染；窗口/Tab 尺寸变化时通过控制消息把新 `rows/cols` 发给后端，后端调用 `TIOCSWINSZ` 同步 PTY 尺寸。
**约束**:
- 支持 ANSI 颜色、光标移动、基本 256 色；UTF-8 输入输出。
- 支持基本键盘快捷键输入（Ctrl+C、Ctrl+D 等），这些是普通键盘事件转发，不被浏览器拦截。

### 4.5. 会话生命周期

**需求**: 刷新浏览器或断线重连时，前端通过当前 URL 所属 workspace（见 §4.10）恢复已有 Tab 列表与各 Tab 的 PTY；若 workspace 为空则自动新建一个 Tab。
**约束**:
- Tab 生命周期由 §4.11 workspace 持久化策略定义；**不再**设"所有 WS 断开后 5 分钟自动清理"。
- 用户登出仅清理本地 session cookie，不终止 workspace 下的任何终端（这些终端由 workspace 模型管理）。

### 4.6. Tab 显示名字与工作目录

**需求**: Tab 标签同时显示用户设定的名字与当前 shell 的工作目录（cwd），随 cwd 变化实时更新。
**动机**: 日常多 Tab 切换时 `Shell 1 / Shell 2` 太难区分；显示 cwd 能一眼看出每个 Tab 在哪个项目下工作，是 tmux 用户的常见诉求。
**约束**:
- 名字来源优先级：用户重命名 > 默认 `Shell N`。
- cwd 以 shell 子进程（PID）的真实工作目录为准，通过读取 `/proc/<pid>/cwd`（或同等方式）周期性刷新，**默认每 2 秒**采样一次；变化时推送给前端。
- 显示格式：`<name> — <cwd 简写>`；cwd 简写规则：`$HOME` 前缀替换为 `~`；总长度超过 40 字符时保留头部前缀 + `…` + 末两段目录（如 `/a/b/.../y/z`）。
- 鼠标悬停（title 属性）显示完整原始 cwd，不做简写。
- shell 进程尚未启动、已退出、或因权限受限（如 macOS SIP）读取 cwd 失败时，cwd 区域留空，仅显示名字；读取失败本身不影响 Tab 的其它功能。

**非目标**: 不跟踪 shell 内执行中的前台命令名（如"vim 里了"），只跟踪 cwd。

### 4.7. 快捷键切换 Tab

**需求**: 前端支持键盘快捷键在 Tab 间切换，不依赖鼠标。
**约束**:
- `Ctrl+Alt+←` / `Ctrl+Alt+→`：切到上一个 / 下一个 Tab；循环切换（首个 Tab 的上一个是最后一个）。
- `Ctrl+Alt+1` … `Ctrl+Alt+9`：跳到第 1…9 个 Tab；对应位置不存在 Tab 时快捷键被忽略（不报错）。
- `Ctrl+Alt+T`：新建 Tab（等价于点击 "+"）；达到 Tab 上限时忽略。
- `Ctrl+Alt+W`：关闭当前 Tab（等价于点击 Tab 上的关闭按钮）。
- 这些快捷键必须在 xterm.js 接管前 **在前端自行拦截**（`preventDefault`），不透传给终端。
- 快捷键组合选 `Ctrl+Alt+*` 而非 `Ctrl+Tab` / `Ctrl+数字`，是因为后者大量被浏览器和 shell 占用。
- 帮助入口：界面上提供一处展示全部快捷键的按钮或弹窗（如 `?` 图标），让用户可查。

**非目标**: 不提供用户自定义快捷键绑定；不支持 macOS `Cmd+` 系映射（Chromium 下仍以 `Ctrl+Alt` 响应）。

### 4.8. Shell 退出自动关闭 Tab

**需求**: 当 Tab 对应的 shell 进程退出（含用户在 shell 中按 `Ctrl-D`、执行 `exit`、kill 等），前端自动关闭该 Tab（等价于点击关闭按钮）。
**动机**: 用户在 shell 里 `Ctrl-D` 的心智模型是"退出这个终端"，此前 shell 虽已退出但 Tab 仍留在 UI 里需要再点一次 × 才消失，不符合本地终端的体验。
**约束**:
- Ctrl-D 本身作为普通键盘事件透传给 PTY，不在前端拦截；真正触发关闭的是后端发来的 `exit` 帧。
- 自动关闭走与手动关闭一致的清理路径（回收 PTY、清理 TabState、若为最后一个 Tab 则新建空 Tab）。
- 关闭前若该 Tab 有未读输出，不阻拦、不二次确认。

### 4.9. 连接状态指示与失焦重连

**需求**: 顶栏把原先的 `connected / closed / error` 文本替换为一个**颜色圆点**指示灯，并在浏览器窗口重新获得焦点时自动尝试重连所有已断开的 Tab WebSocket。
**动机**: 手机或切屏之后 WebSocket 常被浏览器冻结/掐断，用户切回来时希望"自动好"，无需刷新；同时文本提示在手机窄屏下占空间，圆点更紧凑。
**约束**:
- 指示灯颜色：**绿色** = 至少有一个 Tab 的 WS 处于 OPEN；**红色** = 全部 Tab 的 WS 都非 OPEN；**黄色/灰色**（过渡态）= 正在连接/重连。
- 指示灯鼠标悬停（title）显示文字态：`connected` / `disconnected` / `reconnecting`。
- 重连触发事件：`window.focus` 与 `document.visibilitychange`（可见时）；以及 `online` 事件。
- 重连策略：对每个 `readyState !== OPEN` 的 Tab，重新 `new WebSocket(...)`；单次尝试失败不阻塞其它 Tab；同一 Tab 的重连至少间隔 1 秒，避免风暴。
- 后端 Tab 若已被 5 分钟保留策略清理（§4.5），重连时会被 4404 拒绝，前端收到后静默关闭该 Tab（不报红不重试）。

**非目标**: 不做指数退避；不做"断开多久自动放弃"的硬超时（workspace 终端由 §4.11 持久化，没有上限）。

### 4.10. Workspace 与起始目录（URL 指定）

**需求**: 支持通过 URL 查询参数 `?workspace=<id>&cwd=<path>` 把一次浏览器访问绑定到一个具名 workspace，并可选地指定该 workspace 下**新建** Tab 的起始工作目录。
**动机**: 用户日常有多个项目/场景，希望浏览器书签直接打开对应项目目录的终端；并希望能在手机和桌面浏览器之间无缝切换到同一组终端（见 §4.11）。
**约束**:
- `workspace` 合法字符：`[A-Za-z0-9_\-.]`，长度 1–64；越界字符或空字符串视为未指定；未指定时使用名为 `default` 的隐式 workspace。
- `workspace` 大小写敏感：`FOO` 与 `foo` 是两个不同 workspace。
- `cwd` 必须是服务端绝对路径且目录存在；不满足则忽略（fallback 为用户 `$HOME`，仍不可用则 shell 自身默认）。
- `cwd` 仅影响本次 URL 访问之后通过 `+` / `Ctrl+Alt+T` **新建**的 Tab；已存在 Tab 不被迁移、不被重启。
- `cwd` 参数不持久化到 workspace 记录：下一次未带 `cwd` 的访问沿用每个 Tab 已有的 cwd，只有再次带 `cwd` 的访问才改变"新 Tab 起点"。
- workspace 记录在第一次被访问时隐式创建；当其下 Tab 数归零（用户全部手动关闭）时记录被清理。

**非目标**: 不做 workspace 列表页或发现机制（用户自己记 URL）；不做 workspace 级别的重命名或权限隔离。

### 4.11. Workspace 多客户端共享与持久化

**需求**: 同一个 workspace 在不同浏览器 / 设备上打开时，看到**同一组 Tab、同一 PTY 输出**；关闭任一浏览器不会终止 workspace 下的任何终端，只有显式关闭 Tab（§4.3 / §4.8）或服务重启才会终止。
**动机**: 手机 + 电脑多屏幕协同（手机看日志、电脑敲键盘），以及跨时间断点续连（晚上家里开的终端，第二天到公司仍可接上）。
**约束**:
- Tab 列表与每个 Tab 的 PTY 进程按 workspace 维度共享；任一 WS 订阅者都收到所有输出；任一客户端的输入都写入同一 PTY（多人同时打字会冲突，属于预期，不做合并）。
- 任一客户端重命名 Tab、新建/关闭 Tab、resize 都立即对所有 workspace 订阅者生效。
- resize 采用"最小值"合并策略：PTY 的 rows/cols 取所有当前订阅 WS 汇报的 rows/cols 最小值，保证最小客户端也能完整显示。
- 认证不放宽：未登录的客户端即使有 workspace URL 也被重定向到 `/login`，密码门槛不变。
- 登出只清本地 cookie，**不再**终止 workspace 下的终端（修订既有 §4.5 约束）。
- 存储仅内存：服务重启销毁所有 workspace 与其终端（延续 §5.2 的"历史会话不持久化到磁盘"）。

**非目标**: 不做按 workspace 的权限细分（登录后全部可访问）；不做 per-client 视图隔离（输出完全共享，无"每人自己的滚动条"）；不做磁盘持久化；不做冲突检测与输入序列化。

### 4.12. 虚拟键盘粘滞 Ctrl/Alt 作用于下一次字符输入

**需求**: 在虚拟键盘 Ctrl（或 Alt）被点亮的粘滞态下，来自物理键盘或手机软键盘的**下一次**字符输入自动组合为 Ctrl+<char> / Alt+<char> / Ctrl+Alt+<char> 发给终端，然后立即清除粘滞态；用于支持手机端输入 Ctrl+D 等组合键。
**动机**: 手机软键盘没有 Ctrl/Alt 物理键，而粘滞按钮此前只对虚拟键盘自身的 Esc/Tab/方向键生效，导致没法发 Ctrl+D 退出 shell、Ctrl+C 中断命令等基础操作。
**约束**:
- 粘滞态生效时拦截 xterm `onData` 的**首字节**做变换，其余字节原样发送；变换后立即关闭粘滞，恢复普通输入（一次粘滞影响一次输入）。
- Ctrl+<letter>：A–Z 映射到 0x01–0x1a（大小写不敏感）；`@/[/\/]/^/_/?` 映射到 0x00/0x1b/0x1c/0x1d/0x1e/0x1f/0x7f；其它字符（数字、标点）没有定义的控制字符时按普通字符原样发送并清除粘滞。
- Alt+<char>：发送 `\x1b` + 原字符（xterm "Meta sends Escape" 约定）。
- Ctrl+Alt+<char>：先套 Ctrl 规则得到控制字符，再前置 `\x1b`。
- 用户在粘滞高亮状态再次点同一修饰键 → 取消粘滞，下一次输入按普通字符发送。
- 粘滞高亮需保持到一个字符被实际应用或用户取消为止。
- 原虚拟键盘专用键（Esc/Tab/方向/Home/End/PgUp/PgDn）仍走 §4.7 既有编码路径，不受本条影响。

**非目标**: 不做 Shift 粘滞（大小写由软键盘处理）；不做"连续多次粘滞"锁定；不做 Ctrl+方向键等复合修饰（现有虚拟键盘已覆盖）。

## 5. 部署与运行

### 5.1. 反向代理兼容

**需求**: 能在 Nginx 反向代理 + 子路径（如 `https://example.com/wcmux/`）场景下正常工作。
**约束**:
- 所有 HTML 里的资源链接、WebSocket URL 均基于运行时 `base_url` 动态拼接，不写死绝对路径。
- WebSocket 握手遵循 `X-Forwarded-Proto`、`X-Forwarded-Host`（当 `--proxy-headers` 开启时）。

### 5.2. 依赖与运行形态

**需求**: 单文件可运行（`python -m wcmux` 或 `wcmux` 命令），依赖限制在 FastAPI/uvicorn/PTY 库/itsdangerous/psutil 等主流包。
**约束**:
- 支持平台：**Linux**、**macOS**、**Windows 10 1809+**（更旧 Windows 因缺少 ConPTY 不支持）。
- Python 版本要求 `>=3.10`。
- PTY 依赖按平台分装：Unix 使用 `ptyprocess`，Windows 使用 `pywinpty`；上层接口一致。
- cwd 读取统一走 `psutil`，由 `psutil` 内部按平台分派实现。
**非目标**: 暂不打 Docker 镜像、不发 PyPI（后续再议）。
