# wcmux — Web 版 cmux 需求文档

> Revised from spec-20260427.md on 2026-04-29

> 基于 FastAPI 的浏览器端终端多路复用器，支持多 Tab、密码登录，可配置 `base_url` 与 `port`。

## 1. 总体目标

**需求**: 提供一个通过浏览器访问的 cmux/tmux 风格终端工具，用户登录后可在同一页面打开多个终端 Tab，每个 Tab 是一路独立的 shell。
**动机**: 在没有 SSH 客户端或受限网络环境下，也能通过浏览器获得多会话终端；相比只能开单个终端的 Web SSH，Tab 结构让日常多任务运维更顺手。
**非目标**:
- 不实现 tmux 原生的 detach/attach/分屏（pane split）/复制粘贴缓冲区，仅做多 Tab。
- 不支持多用户并发、权限分级；当前只有一套密码、一个用户。
- 不做文件上传下载、不做图形转发。

---

## 2. 运行配置

### 2.1. 启动参数与环境变量

**需求**: 程序启动时可通过命令行参数或环境变量配置以下几项：
- `--port` / `WCMUX_PORT`：HTTP 监听端口，默认 `8022`。
- `--base-url` / `WCMUX_BASE_URL`：部署路径前缀，默认 `/`；用于反向代理下挂载到子路径（如 `/wcmux`）。
- `--password` / `WCMUX_PASSWORD`：登录密码，**必填**，未设置则启动失败并打印明确错误。
- `--shell` / `WCMUX_SHELL`：新 Tab 启动时执行的 shell 可执行文件绝对路径；未设置时按平台选默认：Unix 优先 `$SHELL`，回退 `/bin/bash`；Windows 优先 `%COMSPEC%`，回退 `powershell.exe`。

**约束**:
- 命令行参数优先级高于环境变量。
- `base_url` 规范化：自动补前导 `/`、去掉末尾 `/`（空字符串或 `/` 视为根路径）。
- 所有路由、静态资源、WebSocket 连接地址都必须尊重 `base_url` 前缀；前端页面能在子路径下正常加载。
- `port` 取值范围 `1–65535`，非法值启动失败。

**非目标**: 不支持热加载配置，改了要重启。

### 2.2. 监听地址

**需求**: 默认监听 `0.0.0.0`；可通过 `--host` / `WCMUX_HOST` 覆盖。
**动机**: 默认允许 LAN 访问以便实际使用；但保留回退到 `127.0.0.1` 的能力。

## 3. 密码登录认证

### 3.1. 登录入口

**需求**: 未登录用户访问任何页面/接口都会被重定向到登录页 `/login`（考虑 `base_url` 前缀），登录页提供密码输入框与提交按钮。
**约束**:
- 密码通过 POST `application/x-www-form-urlencoded` 或 JSON 提交，不在 URL query 中传输。
- 密码错误时返回 401 并在页面提示"密码错误"；成功时 set session cookie 并重定向回原目标。

### 3.2. 会话管理

**需求**: 使用服务端签名的 session cookie 记录登录状态，默认有效期 7 天；提供登出入口清除 cookie。
**约束**:
- Cookie 属性必须包含 `HttpOnly`、`SameSite=Lax`；当请求协议为 HTTPS 时自动加 `Secure`。
- Session 秘钥 `--secret-key` / `WCMUX_SECRET_KEY` 可配置；未配置则启动时生成随机 key 并打印警告（重启后 session 失效）。
- WebSocket 握手时必须校验 session cookie，未登录直接拒绝握手（HTTP 401/403）。

### 3.3. 暴力破解保护（基础）

**需求**: 同一 IP 连续 5 次密码错误后，锁定 15 分钟内拒绝所有来自该 IP 的登录尝试。
**动机**: 部署在公网时的最小防护。
**约束**: 锁定状态仅内存保存，进程重启清零；足够作为 L1 防护，不追求严格审计。

### 3.4. 密码加密存储

**需求**: 服务端不以明文形式比对或持久化登录密码；配置接受已加盐的哈希值，运行时用常量时间比较验证；仅提供明文入口时，启动阶段立刻派生哈希并丢弃明文引用。
**动机**: 避免 `ps aux`、systemd `EnvironmentFile`、core dump、日志或环境变量抓取等途径把明文密码泄露给攻击者；也避免配置被 `cat` 误发出去时直接暴露可登录凭证。
**约束**:
- 新增 `--password-hash` / `WCMUX_PASSWORD_HASH`，优先级**高于**明文 `--password` / `WCMUX_PASSWORD`。
- 支持的哈希格式按前缀识别：`$argon2id$...`（推荐）、`$argon2i$...`、`$2a$...` / `$2b$...` / `$2y$...`（bcrypt）。其他前缀启动失败并打印明确错误，不 fallback 到明文。
- 明文入口保留为向后兼容：启动时立即算一次 argon2id 哈希替换使用，**立即解除对明文字符串的引用**（尽 Python 可能），此后该明文不再出现在日志、`repr`、`/proc/<pid>/environ`（配合启动脚本建议用 `unset` 或 `systemd-run --setenv` 的一次性注入）、异常堆栈。
- 同时提供哈希与明文时，以哈希为准、明文被忽略并打印 warning `plaintext password ignored because hash is set`。
- 仅提供明文时启动打印一次 warning `consider providing --password-hash instead of plaintext`。
- 增加子命令 `wcmux hash-password`（`python -m wcmux hash-password`）：从 stdin / 提示符读入密码（不回显），生成 argon2id 哈希打印到 stdout，不写磁盘。
- 哈希校验失败仍按 §3.3 计入 IP 锁定。
- 哈希算法选择由服务端决定（当前 argon2id 参数：`m=64MB, t=3, p=4`），用户只负责给一个合法的哈希字符串，不暴露可调的算法参数。

**非目标**: 不做多用户 / 多密码；不做 API Token；不提供密码迁移工具（变密码就重生成哈希）；不加 HSM / KMS 集成。

## 4. 终端会话与多 Tab

### 4.1. 新建 Tab

**需求**: 页面顶部有"+"按钮，点击后在当前页面新增一个 Tab，并为其在服务端 fork 一路新的 shell 进程（默认 `$SHELL`，fallback `/bin/bash`）。
**约束**:
- 每个 Tab 通过独立的 WebSocket 通道与后端 PTY 双向通信。
- 新建 Tab 时自动切到该 Tab；Tab 名默认 `Shell N`（N 为序号），支持双击重命名。
- 单浏览器页面最多同时存在 **20** 个 Tab，超出时"+"按钮禁用并提示上限。

### 4.2. 切换 Tab

**需求**: 点击 Tab 标签切换到对应终端；未激活的 Tab 在后台保持运行、继续接收输出。
**约束**:
- 切换 Tab 不中断后台 Tab 的 shell 进程。
- 后台 Tab 有新输出时，Tab 标签显示未读提示（如加粗或圆点），切到该 Tab 后清除。

### 4.3. 关闭 Tab

**需求**: 每个 Tab 标签上有关闭按钮；关闭时终止对应 shell 进程（SIGHUP），回收 PTY。
**约束**:
- 关闭最后一个 Tab 后自动打开一个新的空 Tab，不让页面变空。
- 关闭时如有未保存内容，不做额外确认（与本地终端一致）。

### 4.4. 终端渲染与尺寸

**需求**: 前端使用 xterm.js 渲染；窗口/Tab 尺寸变化时通过控制消息把新 `rows/cols` 发给后端，后端调用 `TIOCSWINSZ` 同步 PTY 尺寸。
**约束**:
- 支持 ANSI 颜色、光标移动、基本 256 色；UTF-8 输入输出。
- 支持基本键盘快捷键输入（Ctrl+C、Ctrl+D 等），这些是普通键盘事件转发，不被浏览器拦截。

### 4.5. 会话生命周期

**需求**: 刷新浏览器或断线重连时，前端通过当前 URL 所属 workspace（见 §4.10）恢复已有 Tab 列表与各 Tab 的 PTY；若 workspace 为空则自动新建一个 Tab。
**约束**:
- Tab 生命周期由 §4.11 workspace 持久化策略定义；**不再**设"所有 WS 断开后 5 分钟自动清理"。
- 用户登出仅清理本地 session cookie，不终止 workspace 下的任何终端（这些终端由 workspace 模型管理）。

### 4.6. Tab 显示名字与工作目录

**需求**: Tab 标签同时显示用户设定的名字与当前 shell 的工作目录（cwd），随 cwd 变化实时更新。
**动机**: 日常多 Tab 切换时 `Shell 1 / Shell 2` 太难区分；显示 cwd 能一眼看出每个 Tab 在哪个项目下工作，是 tmux 用户的常见诉求。
**约束**:
- 名字来源优先级：用户重命名 > 默认 `Shell N`。
- cwd 以 shell 子进程（PID）的真实工作目录为准，通过读取 `/proc/<pid>/cwd`（或同等方式）周期性刷新，**默认每 2 秒**采样一次；变化时推送给前端。
- 显示格式：`<name> — <cwd 简写>`；cwd 简写规则：`$HOME` 前缀替换为 `~`；总长度超过 40 字符时保留头部前缀 + `…` + 末两段目录（如 `/a/b/.../y/z`）。
- 鼠标悬停（title 属性）显示完整原始 cwd，不做简写。
- shell 进程尚未启动、已退出、或因权限受限（如 macOS SIP）读取 cwd 失败时，cwd 区域留空，仅显示名字；读取失败本身不影响 Tab 的其它功能。

**非目标**: 不跟踪 shell 内执行中的前台命令名（如"vim 里了"），只跟踪 cwd。

### 4.7. 快捷键切换 Tab

**需求**: 前端支持键盘快捷键在 Tab 间切换，不依赖鼠标。
**约束**:
- `Ctrl+Alt+←` / `Ctrl+Alt+→`：切到上一个 / 下一个 Tab；循环切换（首个 Tab 的上一个是最后一个）。
- `Ctrl+Alt+1` … `Ctrl+Alt+9`：跳到第 1…9 个 Tab；对应位置不存在 Tab 时快捷键被忽略（不报错）。
- `Ctrl+Alt+T`：新建 Tab（等价于点击 "+"）；达到 Tab 上限时忽略。
- `Ctrl+Alt+W`：关闭当前 Tab（等价于点击 Tab 上的关闭按钮）。
- 这些快捷键必须在 xterm.js 接管前 **在前端自行拦截**（`preventDefault`），不透传给终端。
- 快捷键组合选 `Ctrl+Alt+*` 而非 `Ctrl+Tab` / `Ctrl+数字`，是因为后者大量被浏览器和 shell 占用。
- 帮助入口：界面上提供一处展示全部快捷键的按钮或弹窗（如 `?` 图标），让用户可查。

**非目标**: 不提供用户自定义快捷键绑定；不支持 macOS `Cmd+` 系映射（Chromium 下仍以 `Ctrl+Alt` 响应）。

### 4.8. Shell 退出自动关闭 Tab

**需求**: 当 Tab 对应的 shell 进程退出（含用户在 shell 中按 `Ctrl-D`、执行 `exit`、kill 等），前端自动关闭该 Tab（等价于点击关闭按钮）。
**动机**: 用户在 shell 里 `Ctrl-D` 的心智模型是"退出这个终端"，此前 shell 虽已退出但 Tab 仍留在 UI 里需要再点一次 × 才消失，不符合本地终端的体验。
**约束**:
- Ctrl-D 本身作为普通键盘事件透传给 PTY，不在前端拦截；真正触发关闭的是后端发来的 `exit` 帧。
- 自动关闭走与手动关闭一致的清理路径（回收 PTY、清理 TabState、若为最后一个 Tab 则新建空 Tab）。
- 关闭前若该 Tab 有未读输出，不阻拦、不二次确认。

### 4.9. 连接状态指示与失焦重连

**需求**: 顶栏把原先的 `connected / closed / error` 文本替换为一个**颜色圆点**指示灯，并在浏览器窗口重新获得焦点时自动尝试重连所有已断开的 Tab WebSocket。
**动机**: 手机或切屏之后 WebSocket 常被浏览器冻结/掐断，用户切回来时希望"自动好"，无需刷新；同时文本提示在手机窄屏下占空间，圆点更紧凑。
**约束**:
- 指示灯颜色：**绿色** = 至少有一个 Tab 的 WS 处于 OPEN；**红色** = 全部 Tab 的 WS 都非 OPEN；**黄色/灰色**（过渡态）= 正在连接/重连。
- 指示灯鼠标悬停（title）显示文字态：`connected` / `disconnected` / `reconnecting`。
- 重连触发事件：`window.focus` 与 `document.visibilitychange`（可见时）；以及 `online` 事件。
- 重连策略：对每个 `readyState !== OPEN` 的 Tab，重新 `new WebSocket(...)`；单次尝试失败不阻塞其它 Tab；同一 Tab 的重连至少间隔 1 秒，避免风暴。
- **持续重试（页面可见时）**: 单次尝试失败（`onerror` / `onclose` 且非 4401/4404）后，按指数退避继续重试：起始 1s，每次 ×2，上限 30s；成功一次即复位退避计数。
- 触发条件：仅在 `document.visibilityState === "visible"` 时持续重试；页面切到后台（隐藏/失焦）时暂停重试定时器，下一次 focus/visibilitychange 立即恢复（从最短延迟开始）。
- 后端 Tab 若已被 5 分钟保留策略清理（§4.5），重连时会被 4404 拒绝，前端收到后静默关闭该 Tab（不报红不重试）；登录过期 4401 跳登录页。

**非目标**: 不做"断开多久自动放弃"的硬超时（workspace 终端由 §4.11 持久化，没有上限）；不在页面隐藏时重试（省电、避免后台静默失败累计）。

### 4.10. Workspace 由 cwd 标识

**需求**: workspace 的身份**直接由目录路径决定**。URL 查询参数只有 `?cwd=<path>`；服务端把路径**规范化**后作为 workspace id，同一规范化路径在不同浏览器/设备上打开就是同一个 workspace（Tab 与 PTY 共享，见 §4.11）。
**动机**: 用户的日常心智是"把终端开在某个项目目录里"，再额外起一个 workspace 名字既冗余又容易写错；用 cwd 本身做 id，书签就等于"这个目录的共享终端"。
**约束**:
- URL 规则：`?cwd=<绝对路径>`。未提供时 workspace id 取服务端的 `$HOME`（其次 `/`），即存在一个隐式"家目录 workspace"。
- **路径规范化**：`os.path.abspath` + 去除末尾 `/`（根目录 `/` 保留）；大小写敏感（Linux/macOS 语义）；符号链接不解析（保留用户书签里的形式，否则 `~/proj` 和真实的 `/mnt/data/proj` 会被强制合流）。规范化后即 workspace id。
- 同一 workspace 下**新建** Tab 的起始 cwd = workspace id 本身（没有"独立 starting cwd"的二次参数了）。
- `cwd` 不是有效目录（不存在或无读权限）：在 REST 层直接拒绝（`400 invalid cwd`）。不 fallback，避免悄悄把两条书签合到同一个 fallback workspace。
- workspace id 长度超过 4096 字符时拒绝（防止异常书签撑爆内存）。
- workspace 记录在第一次访问时隐式创建；其下所有 Tab 被显式关闭后记录被清理。
- Tab 在生命周期中用户仍可 `cd` 到其它目录，Tab 的实时 cwd（§4.6 显示、§4.10 身份）两者**解耦**：workspace id 只在创建时绑定，之后 Tab 的 cwd 变化只改 Tab 标题，不迁移 Tab 到其它 workspace。

**非目标**: 不再保留独立的 `?workspace=<name>` 参数（旧书签需要改写为 `?cwd=...`）；不做符号链接归一；不做基于 inode 的等价判断；不做 workspace 重命名（想换 id = 开一个新书签重新建 Tab）。

### 4.11. Workspace 多客户端共享与持久化

**需求**: 同一个 workspace（按 §4.10 的 cwd 规则匹配）在不同浏览器 / 设备上打开时，看到**同一组 Tab、同一 PTY 输出**；关闭任一浏览器不会终止 workspace 下的任何终端，只有显式关闭 Tab（§4.3 / §4.8）或服务重启才会终止。
**动机**: 手机 + 电脑多屏幕协同（手机看日志、电脑敲键盘），以及跨时间断点续连（晚上家里开的终端，第二天到公司仍可接上）。
**约束**:
- Tab 列表与每个 Tab 的 PTY 进程按 workspace 维度共享；任一 WS 订阅者都收到所有输出；任一客户端的输入都写入同一 PTY（多人同时打字会冲突，属于预期，不做合并）。
- 任一客户端重命名 Tab、新建/关闭 Tab、resize 都立即对所有 workspace 订阅者生效。
- resize 采用"最小值"合并策略：PTY 的 rows/cols 取所有当前订阅 WS 汇报的 rows/cols 最小值，保证最小客户端也能完整显示。
- 认证不放宽：未登录的客户端即使有 workspace URL 也被重定向到 `/login`，密码门槛不变。
- 登出只清本地 cookie，**不再**终止 workspace 下的终端（修订既有 §4.5 约束）。
- 存储仅内存：服务重启销毁所有 workspace 与其终端（延续 §5.2 的"历史会话不持久化到磁盘"）。

**非目标**: 不做按 workspace 的权限细分（登录后全部可访问）；不做 per-client 视图隔离（输出完全共享，无"每人自己的滚动条"）；不做磁盘持久化；不做冲突检测与输入序列化。

### 4.12. 虚拟键盘粘滞 Ctrl/Alt 作用于下一次字符输入

**需求**: 在虚拟键盘 Ctrl（或 Alt）被点亮的粘滞态下，来自物理键盘或手机软键盘的**下一次**字符输入自动组合为 Ctrl+<char> / Alt+<char> / Ctrl+Alt+<char> 发给终端，然后立即清除粘滞态；用于支持手机端输入 Ctrl+D 等组合键。
**动机**: 手机软键盘没有 Ctrl/Alt 物理键，而粘滞按钮此前只对虚拟键盘自身的 Esc/Tab/方向键生效，导致没法发 Ctrl+D 退出 shell、Ctrl+C 中断命令等基础操作。
**约束**:
- 粘滞态生效时拦截 xterm `onData` 的**首字节**做变换，其余字节原样发送；变换后立即关闭粘滞，恢复普通输入（一次粘滞影响一次输入）。
- Ctrl+<letter>：A–Z 映射到 0x01–0x1a（大小写不敏感）；`@/[/\/]/^/_/?` 映射到 0x00/0x1b/0x1c/0x1d/0x1e/0x1f/0x7f；其它字符（数字、标点）没有定义的控制字符时按普通字符原样发送并清除粘滞。
- Alt+<char>：发送 `\x1b` + 原字符（xterm "Meta sends Escape" 约定）。
- Ctrl+Alt+<char>：先套 Ctrl 规则得到控制字符，再前置 `\x1b`。
- 用户在粘滞高亮状态再次点同一修饰键 → 取消粘滞，下一次输入按普通字符发送。
- 粘滞高亮需保持到一个字符被实际应用或用户取消为止。
- 原虚拟键盘专用键（Esc/Tab/方向/Home/End/PgUp/PgDn）仍走 §4.7 既有编码路径，不受本条影响。

**非目标**: 不做 Shift 粘滞（大小写由软键盘处理）；不做"连续多次粘滞"锁定；不做 Ctrl+方向键等复合修饰（现有虚拟键盘已覆盖）。

### 4.13. 顶栏品牌按钮弹出 Tab 菜单

**需求**: 顶栏左上角 `wcmux` 字样改成可点击按钮；点击弹出下拉菜单，列出当前 workspace 的所有 Tab；点击任意一项激活该 Tab 并关闭菜单。
**动机**: 手机屏幕或 Tab 多时，顶栏水平滚动难看全；下拉菜单能一次纵向浏览全部 Tab，直接跳转。
**约束**:
- 菜单项显示格式沿用 §4.6 的 Tab 标签（`<name> — <cwd_display>`）；悬停 `title` 显示完整 cwd。
- 菜单项顺序与顶栏 Tab 顺序一致；激活的 Tab 在菜单中高亮，未读 Tab 带 §4.2 相同的未读标记。
- 菜单实时随 Tab 列表变化更新（新建/关闭/重命名/cwd 变化通过 §4.11 broadcast 反映）。
- 关闭菜单的触发：点击某一项后、点击菜单外部区域、按 `Esc`。
- 最大高度 ≤ 可视视口 70%，超出时菜单内滚动。
- 菜单不覆盖 toolbar 本身（不挡住 `+` / `?` / logout 等现有按钮）。
- 按钮上"可展开"提示符（caret，如 `▾`）必须使用**缩小 + 降饱和**的内联样式（参考量：`font-size ≈ .7em`、`opacity ≈ .6`、与前文仅留 ~2px 间距），视觉占位不超过半个字符宽；加上 caret 后的 `.brand` 宽度相对纯文字 `wcmux` 的增量 ≤ 6px，不挤占 `#tabs` 可用宽度。
- `.brand` 设 `white-space: nowrap`，在 Tab 挤压时不把 `wcmux` + caret 折成两行（防止 toolbar 视觉增高）。

**非目标**: 不做跨 workspace 列表；不做菜单里搜索/过滤（Tab 最多 20）；不做菜单项内删除/重命名入口（保持只"切换"语义）；不使用独立 padding/border 盒来表达"可点击"（caret 本身已足够提示）。

### 4.14. 虚拟键盘按住连发

**需求**: 虚拟键盘上的**非修饰键**按住不放时自动重复发送：首次触发后 400 ms 开始以约 13 Hz（75 ms 间隔）持续发送，直到手指/鼠标抬起或离开按键。
**动机**: 手机上翻长 log / vim 文件要连续点 PgDn、`↓` 很多次，手酸；按住连发是终端用户直觉。
**约束**:
- 适用范围：`#keypad .kp[data-key]` 类按键（Esc/Tab/Home/End/PgUp/PgDn/←↑↓→、§4.15 新增的 Alt+Backspace）。
- **不**适用：`.kp.mod[data-mod]`（Ctrl/Alt 粘滞切换键）——避免连发导致粘滞态无限翻转。
- 事件驱动：`pointerdown` 开始、`pointerup` / `pointerleave` / `pointercancel` 停止；兼容鼠标与触摸。
- 首次重复延迟 400 ms，之后 75 ms / 次，**硬编码不可配**。
- 首次按下（pointerdown 路径上的第一次发送）应用粘滞 Ctrl/Alt 组合态（§4.12 语义不变）；后续重复发的是裸键字节序列（粘滞已在首次发送后被清掉）。
- 连发期间禁用浏览器长按的默认行为（`preventDefault`），避免弹出系统菜单或选中文本。

**非目标**: 不做每键独立可调的重复率；不做"越按越快"的加速曲线。

### 4.15. 虚拟键 Alt+Backspace

**需求**: 虚拟键盘新增一个专用键（标签 `⌥⌫` 或 `A-⌫`），按下发送字节序列 `\x1b\x7f`（ESC + DEL），对应 bash readline 的 `backward-kill-word`（删除光标前一个单词）。
**动机**: Alt+Backspace 是命令行最常用的编辑组合键之一；手机软键盘的 Backspace 发的是 KeyEvent，不稳定地映射成 `\x08`/`\x7f`，再配合粘滞 Alt 路径在 Android Chromium 下不可靠；给一个**确定性**的专用键最省事。
**约束**:
- 单击发送 `\x1b\x7f`；按住按 §4.14 规则重复发送。
- 粘滞 Alt 点亮时对本键不叠加（已经是 Alt+Del 了，不再前置 ESC）；**粘滞 Ctrl 点亮时忽略 Ctrl**（Ctrl+Backspace 没有稳定约定），按原字节发，并在发送后清粘滞态（与其他虚拟键一致）。
- 放在底部键盘可见区域，不覆盖也不挤掉既有键；具体位置由实现决定（spec 不固化行号）。

**非目标**: 不做额外的"裸 Backspace"虚拟键（软键盘自带 Backspace，够用）；不做 Ctrl+Backspace 字节映射（用户明确要求再加）。

### 4.16. 虚拟键 Ctrl+B（tmux prefix）

**需求**: 虚拟键盘第 2 排从左起第 3 个位置新增一个专用键（标签 `^B` 或 `C-b`），按下发送字节序列 `\x02`（Ctrl-B），即 tmux 默认 prefix 键。第 2 排前 3 格固定为 `PgUp | PgDn | ^B`；后续键顺序见 §4.17。
**动机**: 手机场景下 tmux prefix 是最高频的组合键；走"粘滞 Ctrl + b"路径需要两次点击且依赖软键盘能稳定发 `b`，直接给一个确定性按钮最省事。
**约束**:
- 单击发送 `\x02`；按住按 §4.14 规则重复发送（实际 tmux 很少需要连发，但保持行为一致）。
- 粘滞 Ctrl 点亮时对本键不叠加（已经是 Ctrl-B 了），发送后清粘滞 Ctrl（与其他虚拟键一致）。
- 粘滞 Alt 点亮时忽略 Alt（`\x1b\x02` 无稳定约定），按原 `\x02` 发送并清粘滞 Alt。
- 放在第 2 排"最左起第 3 格"固定位置；其它既有键（方向键 4 个、PgUp/PgDn 2 个）顺序/编码不变。

**非目标**: 不做其他 tmux 子命令（`C-b %`、`C-b "` 等）的一键按钮；不做可配置 prefix（tmux 默认即 C-b，改过的用户走粘滞 Ctrl + 字符键）。

### 4.17. 虚拟键盘字符键 `]` `` ` `` `\`

**需求**: 虚拟键盘第 2 排从左起第 4 格起新增 3 个字符键，依次为 `]`、`` ` ``、`\`；按下原样发送对应 ASCII 字符。第 2 排最终顺序为 `PgUp | PgDn | ^B | ] | `` ` `` | \ | ← | ↑ | ↓ | →`（共 10 格）。
**动机**: Android / iOS 软键盘上这三个键要切 2–3 层符号面板才能找到；tmux 用户常用 `C-b ]`（粘贴缓冲区）、shell 里 `` ` `` 做命令替换、`\` 做换行/转义，给专用键大幅降低手机操作成本。
**约束**:
- 无粘滞时：三键分别发 `]`（0x5d）/`` ` ``（0x60）/`\`（0x5c）。
- **粘滞 Ctrl** 点亮时按 §4.12 语义转换：`]` → `\x1d`（Ctrl-]，telnet escape）、`\` → `\x1c`（Ctrl-\，SIGQUIT）、`` ` `` 无对应控制字符按原字节发；发送后清粘滞 Ctrl。
- **粘滞 Alt** 点亮时前置 `\x1b`（即 ESC + 字符），与其它键一致；发送后清粘滞 Alt。
- 按住按 §4.14 规则重复发送；重复阶段不再叠加粘滞（首次发送时已清）。
- 放在第 2 排第 4–6 格固定位置；不移动 `^B`（§4.16）和原有方向键的相对顺序。

**非目标**: 不做其他常用符号键（`[`、`{`、`|` 等）的批量引入——等后续明确需求再加；不把这三键改成"可配置"。

### 4.18. WebSocket 保活心跳

**需求**: 前端在每条 Tab 的 WebSocket 上周期性发送应用层心跳（`{type:"ping",ts:<epoch_ms>}`），后端回 `{type:"pong",ts:<原 ts>}`；客户端若在预期时间内未收到 pong，主动关闭连接以触发 §4.9 的重连流程。
**动机**: 反向代理（Nginx 默认 60s）、手机运营商 NAT、公司防火墙经常会静默丢弃空闲 WebSocket；浏览器层面无事件触发，`ws.readyState` 仍为 `OPEN`，用户只有在下次敲键时才发现"卡住"。主动心跳把"死连接"转成可观测的 `close`，配合 §4.9 的持续重试就能自愈。
**约束**:
- 心跳间隔：默认 **25 秒**（小于常见代理 30/60s idle 阈值）；仅在 `readyState === OPEN` 时发送。
- 超时判定：发出 ping 后 **10 秒**内未收到对应 `pong` 或任何其它服务端消息 → 视为断连，调用 `ws.close(4000, "heartbeat timeout")` 让 §4.9 接手重连。
- 页面隐藏时（`visibilityState === "hidden"`）暂停心跳定时器，避免后台空耗；恢复可见后立即发一次 ping 探测。
- 后端收到 `{type:"ping"}` 必须立即回 `{type:"pong", ts:<echo>}`；不透传到 PTY，不写 shell 历史。
- 任何服务端消息（`output` / `cwd` / `tabs` / `pong`）都算一次"活性信号"，可复位超时计时器（不必等 pong 才判活）。

**非目标**: 不做 WebSocket 协议层 ping/pong（PING frame）——浏览器不允许 JS 主动发协议 ping，只能走应用层；不把心跳间隔做成可配置（现默认值覆盖绝大多数场景）。

### 4.19. 设备级长效令牌（"记住此设备"）

**需求**: 已登录用户可一键 "记住此设备" 获得一枚长效签名 token；前端存 `localStorage`，下次 cookie 失效时自动用 token 换回 session，无需重新输入密码。同样支持 "忘记此设备"（撤销）。
**动机**: 纯 HTTP 站点的 cookie 在移动浏览器/iOS Safari 经常被悄悄清掉；30d max_age 也无济于事。把"长期信任"显式落到 localStorage，把"是否记住"放回用户自己手里，比和浏览器策略斗智斗勇靠谱。
**约束**:
- Token 用 `itsdangerous.TimestampSigner(secret_key, salt="wcmux-device-token-v1")` 签署 device id；签名层 max_age 90 天，超期不再可用。
- 服务端维护设备登记簿（默认 `~/.local/share/wcmux/devices.json`，结构 `[{id, label, iat, last_seen}]`）；登记簿即"白名单"，删条目立即吊销；可用环境变量 `WCMUX_DEVICES_FILE` 覆盖路径以便测试。
- 三个 REST 端点（均位于 `/api/auth/`）：
  - `POST /issue-device-token`（需 session）：新建设备记录并签发 token，返回 `{token, id}`。
  - `POST /exchange`（无 session）：以 token 换取 session cookie；通过同一 `LockoutRegistry`（与 `/login` 共享桶）做暴力破解节流。
  - `GET /devices` / `DELETE /devices/{id}`（需 session）：列出/吊销。
- 前端：toolbar 一颗按钮在 "记住此设备" / "忘记此设备" 两态间切换；`api()` 在 401 时若 localStorage 存在 token 先尝试一次 `exchange`，失败再跳 `/login`；WS 收到 4401 同样先 exchange 再决定跳登录。
- 安全边界：token 仅是"恢复 session"的钥匙，不直接做请求鉴权（WS 握手仍走 cookie），便于与现有逻辑解耦；旋转 `WCMUX_SECRET_KEY` 立即令所有 token 失效（紧急吊销手段）。

**非目标**: 不做"我的设备"管理 UI（直接编辑 json 或调用 `DELETE /api/auth/devices/{id}`）；不在 WS 层加 token 子协议（间接路径 cookie ↔ token 已够用）；不做基于设备的细粒度权限（单用户工具）。

### 4.20. 终端 URL 可点（xterm web-links）

**需求**: PTY 输出里的 `http(s)://` URL 在 xterm 中渲染为可点击文本；点击/tap 后浏览器以新窗口打开（`_blank` + `noopener,noreferrer`），不导航当前页面、不被弹窗拦截器拦。
**动机**: 之前 wcmux 没装 link addon，URL 只是普通文本——桌面用户可以"选中→复制→粘贴到地址栏"凑合用，**移动端无 hover、无选区可粘贴**，等于完全无法打开链接。这是 §4.19 之后被频繁 demo 时立即踩到的痛点。
**约束**:
- 通过 `@xterm/addon-web-links@0.11.x`（与已用 `xterm@5.5.x` 配套）经 CDN 注入。
- 在 `buildTerminal()` 里每条 Tab 各自 `loadAddon` 一次；插件**可选**（CDN 没加载到时不报错，URL 退化为纯文本，其它功能不受影响）。
- 点击 handler 必须在用户手势栈里同步调 `window.open(uri, "_blank", "noopener,noreferrer")`，确保桌面与移动端都不被弹窗拦截器拦。
- 不修改 PTY 输出、不参与 OSC 解析、不写 replay buffer——纯前端渲染层增强。

**非目标**: 不做 OSC 8 超链接（spec 标准但生态不齐，先靠正则自动识别），不做"链接收件箱"托盘（移动端精准 tap 仍偏弱，但属于独立后续 spec），不去重 / 不限速 / 不持久化历史链接。

### 4.21. 悬浮风格滚动条 / 列宽稳定

**需求**: xterm viewport 用半透明窄滚动条，并永远预留 8px 滚动槽，使终端可用列宽不会因为滚动条出现/消失而抖动。
**动机**: 大屏（4K / 2K）下跑 Claude Code、tmux、less 等长输出时，浏览器原生滚动条出现的瞬间会把可用列数减一两列，已渲染的旧行按旧列宽留着、新行按新列宽折——视觉上像锯齿，体验糟糕；尤其 Edge / Brave（Chromium 系）默认走 classic scrollbar，最容易暴露。
**约束**:
- `.xterm-viewport` 设 `scrollbar-gutter: stable`，确保即便没有可滚动内容，gutter 也常驻预留 → fit addon 计算的列宽全程稳定。
- 视觉：thumb 默认 `rgba(255,255,255,0.18)`，hover `0.36`；track 完全透明；宽 8px。Firefox 走 `scrollbar-width: thin` + `scrollbar-color`，Chromium / WebKit 走 `::-webkit-scrollbar*`，两边覆盖。
- 纯 CSS 改动，不改 JS、不改后端。

**非目标**: 不做"thumb 默认完全透明、滑动时才浮现"（macOS 风格），那是后续可选优化；不为列宽稳定而牺牲滚动功能（不用 `overflow: hidden`）；不针对单个 OS / 浏览器做特殊适配，仅靠两套标准属性兜底。

### 4.22. 文件预览搜索 / Preview 子路由

**需求**: 把 `mdpreview` 的浏览/预览能力以同进程子路由形式并入 wcmux。toolbar 在 `+` 按钮之后增加一个搜索框，输入即时搜 `WCMUX_PREVIEW_ROOT`（默认 `$HOME`）下两层深度的可预览文件，结果点击在新窗口打开 mdpreview 预览页。同时把 `logout` 按钮从 toolbar 移进品牌下拉菜单，给搜索框腾位。
**动机**: 在终端会话里查阅 markdown 笔记 / 代码片段 / 截图 / drawio 图，不需要再开第二个服务、第二张鉴权口；搜索作为入口比左右点目录树或记路径更快，符合"在终端里工作"的肌肉记忆。logout 不是高频操作，藏到下拉里换一片宽度更划算。
**约束**:
- 同进程子路由，复用 wcmux 的 session / 设备 token 鉴权（§4.19），所有 `/api/preview/*`、`/raw/preview/*` 路由前置 `Depends(require_auth)`。mdpreview 自带的 HMAC challenge / `mdpsession` cookie 不引入。
- 路由命名空间：
  - `GET /api/preview/list?path=…` 列目录
  - `GET /api/preview/file?path=…` 取文件（按类型返回 markdown/code/text/image/html/drawio/jsonl）
  - `POST /api/preview/save` 仅允许写 `.drawio`
  - `GET /api/preview/search?q=…&limit=50` 文件名子串搜索
  - `GET /raw/preview/{path}` 原文件流（图片 / iframe / drawio 用）
- 静态资源放在 `static/preview/`（`index.html` / `preview.html` / `style.css`），HTML 内 fetch 全用相对路径（`../../api/preview/…`），保证 `--base-url /wcmux` 反代场景下也对。
- 搜索算法：从 `WCMUX_PREVIEW_ROOT` 起 `os.walk`，深度 ≤ 2（即 root、root/A、root/A/B 三层目录里的文件），跳过隐藏目录（`.` 开头）+ deny-list `node_modules / __pycache__ / .venv / .git / .cache / .local / .npm / .cargo / .rustup / .tox / dist / build`；文件名 / **目录名** case-insensitive 子串匹配；文件扩展名命中 `IMAGE / HTML / DRAWIO / JSONL / MARKDOWN / CODE / TEXT` 白名单（目录无类型限制）；目录命中标 `type:"dir"`，文件按其类型；按 mtime 降序、截到 limit。空 `q` 返 `{"results":[]}`。
- 点击行为按类型分流：文件 → `static/preview/preview.html?path=…`；目录 → `static/preview/index.html?path=…`，进入 mdpreview 文件浏览器在该目录展开。
- 路径穿越：`resolve_path` 在 `resolve()` 之前用 `relative_to(root)` 校验未解析路径必须在 root 下；root 内的符号链接被信任（mdpreview 沿用策略）。
- 前端 UI：
  - 搜索框 `<input type="search" id="preview-search">` 紧跟在 `#new-tab` 之后；下拉 `<div id="preview-results">` 绝对定位于 toolbar 下方。
  - 输入 250ms debounce → `api("GET", "/api/preview/search?q=…")`；401 走现有跳登录路径（§4.19 device-token 也会先尝试 exchange）。
  - 结果项：图标 + 文件名 + 相对路径 + 大小；hover / ↑↓ 高亮；点击 / Enter → `window.open(BASE + "/static/preview/preview.html?path=" + encodeURIComponent(path), "_blank", "noopener,noreferrer")`，调用栈位于用户手势上下文，不被弹窗拦截器拦。Esc / 点空白处关闭。
  - 移动端 `@media (max-width: 600px)` 搜索框 `flex: 1 1 auto`，下拉左右贴边占满 toolbar 下一屏宽。
- logout 移位：toolbar 删 `<form method="POST" action="/logout">`；品牌下拉菜单底部加分隔条 + "logout" 菜单项，点击 `fetch(BASE+"/logout", {method:"POST"})` 后 `window.location = BASE + "/login"`。
- 配置：`WCMUX_PREVIEW_ROOT` 限定**主**根目录，默认 `$HOME`，相对路径以它为锚点；`WCMUX_PREVIEW_EXTRA_ROOTS` 是冒号分隔的额外根列表，让 USB / SD 卡 / NAS 等挂载到 `$HOME` 之外的内容也能被预览/搜索（默认空）。测试用 `/tmp/wcmux-test-preview` + `/tmp/wcmux-test-extra` 隔离。
- `path` 查询参数接受两种形态：
  - **相对**（如 `path=foo/bar.md`）→ 锚到主根；URL 短，common case。
  - **绝对**（如 `path=/media/orangepi/x.md`）→ 校验落入任一 root 下；让 extra root 里的文件可寻址。
- `to_url_path()` 反向映射：在主根下的文件返回相对形式（短）；在 extra root 下的返回绝对形式。这让 search 结果、`<img>` 链接、breadcrumb 都能直接 round-trip。
- `/raw/preview` 改用 `?path=` 查询参数（不再 URL 段），避免绝对路径的前导 `/` 被 HTTP 路径规范化吃掉。
- 搜索：每个 root 独立 `os.walk`，各自 ≤ 2 层深度，结果合并按 mtime 排序。

**非目标**: 不在 wcmux 主页里内嵌预览面板（开新窗口足够，且更安全）；不让 `/api/preview/save` 写非 drawio 文件（markdown 就地编辑保存属于独立后续 spec）；不做"我的搜索历史"、"全文搜索"、"模糊匹配"——文件名子串够用，要更花哨等真撞墙再上 SQLite FTS；不引入 mdpreview 的 `auth.py / setpass.py / login.html` 任何独立登录界面。

### 4.23. 公开分享 / Share

**需求**: 已登录用户可在 preview 页右上角点 "🔗 分享" 把当前文件做成一条公开 URL，无需登录即可访问；URL 嵌入日期 + 文件名 slug，便于在浏览器历史、聊天消息中辨识；过期可选 1d / 7d / 1mo / 3mo / **1y（默认）**/ 3y / 永不；支持立即撤销与"我的分享"列表管理。
**动机**: 在终端 + 文件预览体系里日常会想"把这条笔记甩给同事看一眼"。当前所有 `/api/preview/*` 都需 session，发出去对方看不了；自己再每次截图、贴文又损失 markdown / 代码 / 图的版式。一条不需要登录、有过期、可撤销的公开链接，跟 Google Docs / GitHub Gist 的"share with link"是同一档常识体验。
**约束**:
- URL 形态：`<base_url>/share/<YYYY-MM-DD>/<slug>-<id12>`。
  - `<YYYY-MM-DD>` = 创建日期，便于按时间定位。
  - `<slug>` = 文件名 ascii kebab，cap **200 字符**（filename 长保留，URL 仍 < 250 char）。
  - `<id12>` = `secrets.choice(string.ascii_letters + string.digits)` × 12 ≈ 71 bit 熵；只 alphanum 不含 `-`/`_`，让"slug-id"末段 12 字符可被 `rsplit('-',1)` 唯一解析。
- 登记簿：`~/.local/share/wcmux/shares.json`（`WCMUX_SHARES_FILE` 覆盖），结构 `{shares: [{id, date, slug, label, source_url_path, source_real_path, created_at, expires_at, view_count, last_viewed, assets, assets_skipped}]}`。删条目 = 立即吊销。
- 过期：`expires_at` 存 epoch 秒；`0` = 永不过期。lookup 时 `now > expires_at and expires_at != 0` → 410 Gone。
- 鉴权：
  - `/api/share*`（CRUD）走 `Depends(require_auth)`。
  - `/share/<date>/<seg>` 与 `/share/<date>/<seg>/a` 与 `/share/<date>/<seg>/raw` **公开**，凭 12 字符 id 即过；`LockoutRegistry` 与 `/login` 共用桶——错 5 次该 IP 锁 15 分钟。`<date>` 必须与登记簿中的日期匹配，否则 404（防误传 / 过期截屏复用）。
- 文件类型支持（MVP）：markdown / code / text / image。html / jsonl / drawio 返 415。
- 资产打包（仅 markdown）：扫源 markdown 中 `![](...)` 与 `<img src=...>`，跳过 `http://` / `https://` / `data:` / `#` / `mailto:` / `javascript:`；剩下的候选要**双重满足**：未解析路径 + `resolve()` 后真实路径都落入任一 `preview_roots` 下；命中 `IMAGE_EXTS`、不是隐藏文件、是常规文件；累计配额 ≤ **50 张** / **100 MiB**。超额 / 越界条目进 `assets_skipped`，前端 dialog 黄字警告但不阻断分享。
- 渲染：服务端拼**完整自包含 HTML**，CSS 内联，不引外部 JS / 字体；
  - markdown 走 `markdown` lib（与 `/api/preview/file` 同一套 extensions），代码块用 `pygments` 服务端高亮（`HtmlFormatter(noclasses=True)` 内联样式，免外部 stylesheet）；
  - code 走 `pygments` 全文件高亮；
  - text `<pre>` 显示；
  - image 顶层 `<img>` 指向 `/share/<date>/<slug-id>/raw?path=<source_url_path>`。
- HTTP 安全头：所有 `/share/*` 响应带
  - `Cache-Control: private, no-store`
  - `X-Frame-Options: DENY`、`frame-ancestors 'none'`
  - `Content-Security-Policy: default-src 'none'; img-src 'self' data:; style-src 'self' 'unsafe-inline'; script-src 'none'`
  - `Referrer-Policy: no-referrer`、`X-Content-Type-Options: nosniff`
- UI：
  - preview.html 顶栏右侧加 `🔗 分享` 按钮，开 `<dialog>` 选过期 / 备注，[生成链接] → 显示 URL + [复制] + [撤销]，含 `assets_skipped` 黄字。
  - brand 下拉新增 "我的分享" → 打开 `/static/preview/shares.html`，列出全部 share、含访问数 / 最近访问相对时间 / [复制] / [撤销]。
- 撤销层级：删 share 立即返 404；旋转 `WCMUX_SECRET_KEY` 不影响 share（id 是随机的，不是签名 token）。

**非目标**: 不做密码保护的 share（用过期 + 撤销 + 不可枚举的 id 已经够私有工具档位）；不做"分享给指定用户"（无多用户模型）；不做编辑保存版本的分享（freeze on share，不跟随源文件后续修改——本期是简单形态，后续若做 "live share" 单独 spec）；不在 share 页面跑任何 JS（CSP `script-src 'none'` 是硬约束，所以代码高亮只能靠 pygments 服务端，不用 highlight.js）；MVP 暂不支持 html / jsonl / drawio 类型（jsonl 体量大、html 引 iframe 风险面、drawio 依赖 embed.diagrams.net 跨域）。

## 5. 部署与运行

### 5.1. 反向代理兼容

**需求**: 能在 Nginx 反向代理 + 子路径（如 `https://example.com/wcmux/`）场景下正常工作。
**约束**:
- 所有 HTML 里的资源链接、WebSocket URL 均基于运行时 `base_url` 动态拼接，不写死绝对路径。
- WebSocket 握手遵循 `X-Forwarded-Proto`、`X-Forwarded-Host`（当 `--proxy-headers` 开启时）。

### 5.2. 依赖与运行形态

**需求**: 单文件可运行（`python -m wcmux` 或 `wcmux` 命令），依赖限制在 FastAPI/uvicorn/PTY 库/itsdangerous/psutil 等主流包。
**约束**:
- 支持平台：**Linux**、**macOS**、**Windows 10 1809+**（更旧 Windows 因缺少 ConPTY 不支持）。
- Python 版本要求 `>=3.10`。
- PTY 依赖按平台分装：Unix 使用 `ptyprocess`，Windows 使用 `pywinpty`；上层接口一致。
- cwd 读取统一走 `psutil`，由 `psutil` 内部按平台分派实现。
**非目标**: 暂不打 Docker 镜像、不发 PyPI（后续再议）。
