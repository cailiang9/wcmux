# wcmux — 验证 Checklist

> Revised from testing-20260424.md on 2026-04-25

与 `spec-20260424.md` 一一对应；每条 spec 至少覆盖一个正常路径 + 一个边界 + 一个错误路径。

## 1. 总体目标
对应: spec.md §1

- [ ] 用默认配置启动后，浏览器访问根路径能看到登录页（非 500/404）
- [ ] 登录成功后页面默认至少有一个可交互 Tab，输入 `echo hello` 能看到 `hello` 输出
- [ ] 文档/README 明确写了"不支持分屏 / 不支持多用户"等非目标，避免误用

---

## 2.1. 启动参数与环境变量
对应: spec.md §2.1

- [ ] `wcmux --port 9000 --password pw` 启动后，`curl http://127.0.0.1:9000/login` 返回 200
- [ ] 仅设置环境变量 `WCMUX_PORT=9001 WCMUX_PASSWORD=pw wcmux` 启动后监听 9001 端口
- [ ] 同时给命令行 `--port 9000` 和环境 `WCMUX_PORT=9001`，实际监听 9000（命令行优先）
- [ ] 未提供 `--password` 且未设置 `WCMUX_PASSWORD` 时，启动失败且 stderr 打印包含 "password" 的明确错误，退出码非 0
- [ ] `--base-url wcmux`、`--base-url /wcmux`、`--base-url /wcmux/` 三种写法都被规范化为 `/wcmux`，登录页 URL 为 `/wcmux/login`
- [ ] `--base-url /` 或空字符串情况下，登录页 URL 为 `/login`，不出现 `//login`
- [ ] `--port 70000`（越界）或 `--port abc`（非数字）启动失败，错误信息指明 port 非法
- [ ] 未设 `--shell`、`$SHELL` 有值时新 Tab 跑的是 `$SHELL` 指向的 shell（`echo $0` / `$PSVersionTable` 验证）
- [ ] 未设 `--shell`、Unix 下 `$SHELL` 为空时回退到 `/bin/bash`；Windows 下 `%COMSPEC%` 为空时回退到 `powershell.exe`
- [ ] `--shell /bin/zsh`（或 Windows `--shell C:\Windows\System32\cmd.exe`）启动后新 Tab 确实用该 shell
- [ ] `--shell /not/exist/sh` 指向不可执行路径时，新建 Tab 请求返回错误（或后端记录错误、Tab 标记为 exit），不崩整个服务

## 2.2. 监听地址
对应: spec.md §2.2

- [ ] 默认启动后 `ss -lntp` 或 `netstat` 能看到进程监听 `0.0.0.0:<port>`
- [ ] `--host 127.0.0.1` 启动后，从另一台机器访问该端口连接被拒

## 3.1. 登录入口
对应: spec.md §3.1

- [ ] 未登录时访问 `/`、`/terminal`、`/api/tabs` 等路径都被 302 重定向到 `/login`（含 `base_url` 前缀）
- [ ] 登录页 POST 正确密码后重定向到登录前原目标（如 `/terminal`），并带上 session cookie
- [ ] 登录页 POST 错误密码后返回 401，页面仍显示登录表单并提示"密码错误"
- [ ] 密码参数只允许 POST body，`GET /login?password=pw` 不能完成登录

## 3.2. 会话管理
对应: spec.md §3.2

- [ ] 登录成功后 cookie 响应头包含 `HttpOnly; SameSite=Lax`
- [ ] 在 HTTPS（反向代理）下 cookie 包含 `Secure`；在纯 HTTP 本地访问不带 `Secure`
- [ ] 登出后 cookie 被清除（`Set-Cookie: ...=; Max-Age=0` 或等价），再次访问受保护资源被重定向到 `/login`
- [ ] WebSocket 握手时伪造/缺失 cookie 会被服务端以 401/403 关闭连接，前端拿不到终端输出
- [ ] 未配置 `WCMUX_SECRET_KEY` 启动时 stderr 打印警告；重启后旧 cookie 失效（需重新登录）
- [ ] 配置 `WCMUX_SECRET_KEY=xxx` 后，重启进程旧 cookie 仍然有效

## 3.3. 暴力破解保护
对应: spec.md §3.3

- [ ] 同一 IP 连续 5 次错误密码后，第 6 次即使密码正确也被拒，返回锁定错误
- [ ] 锁定期间持续 15 分钟拒绝，15 分钟后正确密码可登录
- [ ] 不同 IP 之间互不影响：IP-A 被锁定时 IP-B 仍可正常登录
- [ ] 进程重启后锁定状态清零（符合"仅内存保存"设计）

## 3.4. 密码加密存储
对应: spec.md §3.4

- [ ] `wcmux hash-password` 提示输入密码（不回显），stdout 输出形如 `$argon2id$...` 的哈希
- [ ] 用上一步产出的哈希通过 `--password-hash '$argon2id$...'` 启动后，用原明文能登录；用别的明文返回 401
- [ ] `--password-hash '$2b$12$...'`（bcrypt 格式）启动，正确明文能登录
- [ ] `--password-hash 'plaintext-not-a-hash'` 启动直接失败，退出码非 0 且 stderr 提示不支持的哈希格式
- [ ] 同时设置 `WCMUX_PASSWORD=aa` 和 `WCMUX_PASSWORD_HASH='$argon2id$...of bb'`，启动后 stderr 有 warning，实际用 `bb` 登录成功、`aa` 失败
- [ ] 仅设 `WCMUX_PASSWORD=xx` 启动，stderr 一次性打印 `consider providing --password-hash` warning；登录用 `xx` 可通过
- [ ] 仅设明文启动后 `grep -a xx /proc/<pid>/environ` **还是能看到**（为环境变量导入路径的固有限制，记录作为非阻断项）；但启动日志不含明文、`ps aux` 不含明文（因为 CLI 侧用 `--password-hash` 正常情况下不传明文）
- [ ] 输错 5 次密码（明文或哈希模式均然）后第 6 次正确密码被 IP 锁定拒绝（§3.3 仍生效）

## 4.1. 新建 Tab
对应: spec.md §4.1

- [ ] 点击"+"后页面新增一个 Tab，自动激活，后端能看到新的 shell 子进程（`ps` 可见）
- [ ] Tab 名默认为 `Shell 1`、`Shell 2` 递增；双击重命名后名字保留并显示
- [ ] 新 Tab 环境变量 `$SHELL` 无值时使用 `/bin/bash`；可通过 `echo $0` 验证
- [ ] 连续点击"+"开到第 20 个后，"+"按钮 disabled，提示达到上限
- [ ] 达上限后关闭一个 Tab，再点"+"能继续新建

## 4.2. 切换 Tab
对应: spec.md §4.2

- [ ] 在 Tab A 运行 `sleep 5 && echo A`，立刻切到 Tab B；5 秒后切回 A 能看到 `A` 输出
- [ ] Tab B 后台产生输出时，Tab B 标签出现未读标记；切到 B 后未读标记消失
- [ ] 当前激活 Tab 的未读标记永不出现

## 4.3. 关闭 Tab
对应: spec.md §4.3

- [ ] 关闭 Tab 后对应 shell 进程 PID 消失（收到 SIGHUP 被终止）
- [ ] 关闭最后一个 Tab 时自动打开一个新空 Tab，页面不会变成空白
- [ ] 多 Tab 下关闭非激活 Tab 不影响激活 Tab 内容

## 4.4. 终端渲染与尺寸
对应: spec.md §4.4

- [ ] `ls --color=auto` 输出颜色正确显示
- [ ] `vim` / `htop` / `less` 能正常进入全屏界面并正确退出
- [ ] 改变浏览器窗口大小后 `tput cols` `tput lines` 与可见尺寸一致
- [ ] 中文（UTF-8）输入输出不乱码：`echo 你好` 正常显示
- [ ] Ctrl+C 能中断 `sleep 100`；Ctrl+D 在空行能结束一个交互式 python 会话

## 4.5. 会话生命周期
对应: spec.md §4.5

- [ ] 开 3 个 Tab 后刷新浏览器，3 个 Tab 仍在，内容历史保留
- [ ] 关闭浏览器标签页任意时长（含 > 5 分钟）再打开同 URL，Tab 与 shell 仍在（由 §4.11 持久化保证）
- [ ] 点击"登出"后仅 session cookie 被清，workspace 下 shell 进程仍在运行；重新登录并打开同 URL 能接着用
- [ ] 空 workspace（无任何 Tab）访问时自动新建一个 Tab，页面不会停留在空白状态

## 4.6. Tab 显示名字与工作目录
对应: spec.md §4.6

- [ ] 新建 Tab 默认显示 `Shell 1 — ~`（或 `Shell 1 — <启动时 cwd 简写>`）
- [ ] 在 Tab 内执行 `cd /tmp` 后 2 秒内标签更新为 `Shell 1 — /tmp`；再 `cd -` 回来标签恢复
- [ ] cwd 位于 `$HOME` 下（如 `$HOME/projects`）时显示为 `~/projects`，非 `/home/user/projects`
- [ ] cwd 长度超过 40 字符时显示为 `<前缀>…/<倒数第二段>/<最后一段>`，鼠标悬停 title 显示完整原始路径
- [ ] 双击重命名 Tab 后，标签格式为 `<新名字> — <cwd>`；cwd 继续随 `cd` 更新
- [ ] shell 进程异常退出（如 `exit` 后未重连）时标签只剩名字，cwd 区域为空、不显示 `undefined`
- [ ] cwd 无变化时不产生多余 WS 推送（抽查 devtools，2 秒周期内没有重复相同 cwd 的消息）
- [ ] 模拟 `psutil.Process.cwd()` 抛 `AccessDenied`（如 mock 或在受限 macOS 进程上验证）时 Tab 标签只剩名字、其它功能不受影响，后端日志打印一次 warning 但不刷屏

## 4.7. 快捷键切换 Tab
对应: spec.md §4.7

- [ ] 打开 3 个 Tab 停在第 2 个，按 `Ctrl+Alt+→` 切到第 3 个；再按一次切到第 1 个（循环）
- [ ] `Ctrl+Alt+←` 反向切换，行为对称
- [ ] 打开 3 个 Tab 按 `Ctrl+Alt+2` 跳到第 2 个；按 `Ctrl+Alt+5`（不存在）无任何反应且控制台无报错
- [ ] `Ctrl+Alt+T` 新建一个 Tab 并激活；连续按到达 20 上限后继续按无反应
- [ ] `Ctrl+Alt+W` 关闭当前 Tab，行为与点击关闭按钮一致
- [ ] 焦点在 xterm.js 终端里时按快捷键也生效，且不会把字符写进终端（不出现残留）
- [ ] 界面上能找到快捷键说明入口（按钮 / `?` 图标），点开能看到全部 5 组快捷键

## 4.8. Shell 退出自动关闭 Tab
对应: spec.md §4.8

- [ ] 在 bash Tab 里按 `Ctrl-D`（空行）后 shell 退出，前端 Tab 在 1s 内自动消失
- [ ] 在 Tab 里执行 `exit` 也触发同样的自动关闭
- [ ] 关闭最后一个 Tab 后页面自动打开一个新的空 Tab（与手动关闭一致）
- [ ] 后端 shell 子进程 PID 被回收（`ps` 不再可见）
- [ ] Ctrl-D 在交互程序（如 `cat` 或 `python`）里仅影响该程序自身，shell 未退出时 Tab 不关闭

## 4.9. 连接状态指示与失焦重连
对应: spec.md §4.9

- [ ] 页面正常运行时顶栏显示**绿色**圆点，悬停 title 为 `connected`
- [ ] 手动断网 / 关掉后端后，红点出现，title 为 `disconnected`
- [ ] 重新联网（或后端恢复）后无需刷新，圆点自动变绿并恢复可输入
- [ ] 切到其它浏览器 Tab 再切回（触发 `visibilitychange`），前端对已断开的 Tab 发起重连
- [ ] 手机上锁屏/解锁（`window.focus` 触发）后自动重连成功
- [ ] 后端 5 分钟保留期已过的 Tab 重连时被 4404 拒绝，前端静默关闭该 Tab，不留红点也不无限重试
- [ ] 同一 Tab 连续重连间隔 ≥ 1 秒（devtools 网络面板肉眼验证，不出现 10Hz 重连风暴）
- [ ] **持续重试**：停掉后端，前端（页面保持可见）能按 1s / 2s / 4s / 8s / 16s / 30s / 30s… 的退避持续发起重连尝试（devtools 网络面板可见），不因单次失败就放弃
- [ ] 重连成功一次后再次断开，退避从 1 秒重新开始（不沿用上次的 30 秒上限）
- [ ] 页面切到后台（`visibilityState === "hidden"`）时，退避定时器暂停，devtools 不再看到新的 WS 连接请求；切回可见后立即从 1 秒起重试

## 4.10. Workspace 由 cwd 标识
对应: spec.md §4.10

- [ ] 访问 `/?cwd=/tmp`，新建 Tab 的初始目录为 `/tmp`（`pwd` 输出 `/tmp`，Tab 标签显示 `/tmp`）
- [ ] 不带 `?cwd=` 访问 `/`，workspace id 等于服务端 `$HOME`；新 Tab 在 `$HOME` 起；同目录的另一浏览器（无参数）看到同一 Tab
- [ ] `?cwd=/tmp/` 与 `?cwd=/tmp` 规范化成同一 workspace（末尾斜杠忽略）
- [ ] `?cwd=/Tmp` 与 `?cwd=/tmp` 是不同 workspace（大小写敏感）
- [ ] `?cwd=/not/exist/here` 返回 400 `invalid cwd`，前端不创建 Tab
- [ ] 两端浏览器同打开 `?cwd=/tmp`，A 端新建 Tab 后 B 端 1s 内看到该 Tab；共用同一 PTY
- [ ] 在某 Tab 内 `cd /var` 后，Tab 标签显示 `/var`，但 Tab 仍属于原先 `/tmp` workspace（不会"跳到" `/var` 的 workspace 下）
- [ ] 关掉 `/tmp` workspace 下所有 Tab 后，workspace 记录被 GC；再访问是新的空 workspace
- [ ] 旧书签 `?workspace=proj1&cwd=/tmp` 中的 `workspace` 参数被忽略；只 `cwd=/tmp` 生效（符号链接不解析仍保持用户书签形式）

## 4.11. Workspace 多客户端共享与持久化
对应: spec.md §4.11

- [ ] 两个浏览器打开同一 `?workspace=X`，任一端新建 Tab，另一端 1s 内也看到该 Tab 出现
- [ ] 任一端在某 Tab 输入命令，另一端同一 Tab 能看到相同输出
- [ ] 任一端关闭一个 Tab，另一端该 Tab 同步消失
- [ ] 一端浏览器整个关掉后，另一端继续看到终端运行不中断；首端重开同 URL 能接上
- [ ] 未登录的浏览器访问 `?workspace=X` 仍被 302 到 `/login`，不绕过密码
- [ ] POST `/logout` 后该浏览器 session cookie 被清，但 workspace 下 `ps` 仍能看到 shell 进程
- [ ] 两端同一 Tab 的 xterm 窗口大小不同时，PTY 的 `tput cols/lines` 取两者最小值
- [ ] 服务重启后所有 workspace 及其终端被销毁（符合仅内存存储）

## 4.12. 粘滞 Ctrl/Alt 作用于下一次字符输入
对应: spec.md §4.12

- [ ] 点亮虚拟键盘 `Ctrl`，在 bash 中敲小写 `d` → shell 退出（即收到 0x04），Ctrl 高亮随后自动熄灭
- [ ] 点 `Ctrl` + 敲 `c` 在 `sleep 100` 期间 → 进程被中断（0x03）
- [ ] 点 `Alt` + 敲 `f` 在 bash 中 → 光标向右跳一个单词（readline 收到 ESC f）
- [ ] 点 `Ctrl` 再点 `Alt` 都点亮后，敲 `m` → 终端收到 `\x1b\x0d`（ESC + Ctrl-M）
- [ ] 点亮 `Ctrl` 后敲 `5`（无对应控制字符）→ 终端收到普通 `5`，Ctrl 粘滞也被清除
- [ ] 连点 `Ctrl` 两次（再次点取消）后敲 `d` → 收到普通 `d`，shell 不退出
- [ ] 粘滞高亮持续到实际应用或再次点击为止（不会超时自动熄灭）
- [ ] 大小写不敏感：`Ctrl+D` 与 `Ctrl+d` 都发 0x04

## 4.13. 顶栏品牌按钮弹出 Tab 菜单
对应: spec.md §4.13

- [ ] 点击左上角 `wcmux` 按钮，下拉菜单出现，列出当前 workspace 下所有 Tab；项顺序与顶栏一致
- [ ] 在菜单里点某一项，对应 Tab 被激活，菜单随之关闭
- [ ] 菜单外部点击任意处关闭菜单；焦点在菜单时按 `Esc` 也关闭
- [ ] 新建或关闭 Tab 时菜单内列表实时更新（另一浏览器触发同 workspace 的变化也会反映）
- [ ] 当前激活 Tab 在菜单里高亮；有未读的非激活 Tab 带未读标记
- [ ] Tab 数超过视口时菜单内可滚动，且不遮挡 `+` / `?` / logout 按钮
- [ ] 菜单项 hover `title` 显示 Tab 的完整 cwd（未简写）
- [ ] caret（▾）字号 ≈ `.7em` 且 `opacity` ≈ `.6`；devtools 量 `.brand` 宽度，加 caret 前后差值 ≤ 6px
- [ ] 多开 Tab 把顶栏挤到刚好溢出时，`wcmux` + caret 仍单行显示，toolbar 高度保持 36px 不变高
- [ ] 窄屏（≤ 360px）横向容纳不下所有 Tab 时，`.brand` 不被压缩到换行；`#tabs` 按既有 `overflow-x: auto` 滚动

## 4.14. 虚拟键盘按住连发
对应: spec.md §4.14

- [ ] 按住 `PgDn` 不动 ~2 秒，终端收到 ≥ 10 次 `\x1b[6~`（可在 `less` / `man` 文档里肉眼看到滚动连续）
- [ ] 按下 `↓` 后 400 ms 内不应出现第二次（首次延迟），之后每约 75 ms 一次
- [ ] 抬起手指/鼠标后立即停止，没有残留的"拖尾发送"
- [ ] 按住 `Ctrl`（修饰键）不连发——第二次不会翻转回关闭状态
- [ ] 点击 `↑` 一次（按下即抬）只发一次，不触发重复路径
- [ ] 先点亮 `Ctrl` 再按住 `c`（假想的物理键；这里测 sticky + 虚拟方向键组合）：首次发 Ctrl+组合，后续发裸键
- [ ] 长按期间浏览器不弹系统菜单、也不出现系统级文字选中

## 4.15. 虚拟键 Alt+Backspace
对应: spec.md §4.15

- [ ] 点击 `⌥⌫` 键，终端收到字节序列 `\x1b\x7f`；bash 光标前一个单词被删（`hello world` → `hello `）

- [ ] 粘滞 `Alt` 点亮后再点 `⌥⌫`，发送仍是 `\x1b\x7f`（不变成 `\x1b\x1b\x7f`），粘滞 Alt 被清
- [ ] 粘滞 `Ctrl` 点亮后再点 `⌥⌫`，发送仍是 `\x1b\x7f`（Ctrl 被忽略），粘滞 Ctrl 被清
- [ ] `⌥⌫` 按键在界面上可见，不遮挡现有 `+` / 方向键 / logout 等

## 4.16. 虚拟键 Ctrl+B（tmux prefix）
对应: spec.md §4.16

- [ ] 第 2 排虚拟键盘从左起第 3 格出现 `^B`（或 `C-b`）按钮；前 3 格为 `PgUp | PgDn | ^B`（完整顺序见 §4.17）
- [ ] 在 tmux 里点 `^B` 再点 `c` → 新开 tmux 窗口（等价于 `Ctrl-B c`）
- [ ] 未跑 tmux 时，在空 bash 行点 `^B` → `cat -v` / `showkey -a` 能看到 `^B` / `0x02`
- [ ] 按住 `^B` 按 §4.14 节奏连发（400 ms 首发、~75 ms 重复），终端看到 ≥ 5 次 `\x02`
- [ ] 粘滞 `Ctrl` 点亮后再点 `^B`，发送仍是单字节 `\x02`（不重复叠加 Ctrl），粘滞 Ctrl 被清除
- [ ] 粘滞 `Alt` 点亮后再点 `^B`，发送仍是 `\x02`（未前置 ESC），粘滞 Alt 被清除
- [ ] 新增按键不把既有方向键 / PgUp / PgDn 推出视口；窄屏下第 2 排按 `.kp-row` 的 `overflow-x: auto` 可横向滚动

## 4.17. 虚拟键盘字符键 `]` `` ` `` `\`
对应: spec.md §4.17

- [ ] 第 2 排完整顺序为 `PgUp | PgDn | ^B | ] | `` ` `` | \ | ← | ↑ | ↓ | →`（共 10 格）
- [ ] 无粘滞时点击三键，终端分别收到 `]` / `` ` `` / `\`（用 `cat` 或在 bash 命令行回显验证）
- [ ] 粘滞 `Ctrl` 点亮后点 `]` → 终端收到 `\x1d`（可用 `cat -v` 看到 `^]`）；粘滞 Ctrl 被清
- [ ] 粘滞 `Ctrl` 点亮后点 `\` → 终端收到 `\x1c`（`cat -v` 看到 `^\`；若前台是可中止前台任务，触发 SIGQUIT 产生 coredump 提示）；粘滞 Ctrl 被清
- [ ] 粘滞 `Ctrl` 点亮后点 `` ` `` → 终端收到普通 `` ` ``（无 Ctrl 映射）；粘滞 Ctrl 被清
- [ ] 粘滞 `Alt` 点亮后点 `]` → 终端收到 `\x1b]`（ESC + `]`）；粘滞 Alt 被清
- [ ] 按住任一字符键 ≥ 2 秒，终端看到 ≥ 10 次重复字符；按 §4.14 的 400 ms / 75 ms 节奏
- [ ] 三键新增后既有 `^B`、方向键、PgUp/PgDn 的点击行为全部不变（§4.16 / §4.14 的 checklist 能重跑通过）

## 4.18. WebSocket 保活心跳
对应: spec.md §4.18

- [ ] 页面保持打开 2 分钟，devtools 的 WS 帧面板能看到每 ~25 秒一次的 `{"type":"ping",...}` 发出 + `{"type":"pong",...}` 回包
- [ ] `ping` 不写进终端（xterm 可见区域里搜不到 `ping` / `pong` 字样）
- [ ] 在 Nginx 反代且把 `proxy_read_timeout` 调成 `20s`（小于心跳间隔）会持续断连重连——改回 `3600s` 后稳定不断
- [ ] 用 `iptables -I OUTPUT -p tcp --dport 8022 -j DROP` 模拟静默丢包，10 秒内客户端应主动关闭 WS（devtools 看到 close 事件、code `4000` 或相近），随后进入 §4.9 的重连退避；删掉 iptables 规则后自动恢复
- [ ] 页面切到后台时 ping 定时器暂停（切回前台后第一条帧距离上次超过 25 秒也不报错），切回可见后立即发一次 ping
- [ ] 后端其它业务消息（`output` / `cwd` / `tabs`）到达时，超时计时器应被复位：即使 pong 丢失但有别的流量，不误判为断连
- [ ] 客户端每秒打满输入的压力场景下心跳仍按节奏发出，不被 xterm 输入线程饿死
